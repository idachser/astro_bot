# План: переход с парсинга astronomy.com на iCal-фид in-the-sky.org

## Цель

Заменить хрупкий скрейпинг HTML-статьи «Sky This Week» на структурированный
источник — iCal-календарь событий in-the-sky.org:

```
https://in-the-sky.org/newscalyear_ical.php?year=<YYYY>&maxdiff=7
```

Фид отдаёт ~500 событий в год (фазы Луны, соединения, противостояния,
покрытия, метеорные потоки, затмения) с точным UTC-временем, заголовком,
описанием и ссылкой. Данные © Dominic Ford, использование — некоммерческое
с обязательной атрибуцией.

## Этап 7. Переход на собственный сервис событий (skyevents)

Сервис `skyevents` (отдельный репозиторий) считает события сам
(Skyfield + эфемериды JPL, офлайн) и отдаёт их по HTTP API (`/v1/...`).
Он деплоится самостоятельно; бот знает о нём только `EVENTS_API_URL`
в `.env` — в docker-compose бота сервис не входит. Переход снимает
зависимость от неофициального фида и лицензионное ограничение
атрибуции, а тексты становятся доступны на русском (`lang=ru`).

- [ ] Источник данных: HTTP-клиент к `GET /v1/events` вместо
      `services/ics_feed.py`, возвращающий те же словари
      `{uid, dt_utc, summary, description, url}` — `sync_events()` и всё
      ниже не меняются. Учесть пустой `url` (из фида он приходил всегда).
      Промежуточный вариант без правки клиента: у сервиса есть
      `GET /v1/calendar.ics?year=` в формате фида — достаточно вынести
      `ICS_FEED_URL` из `config.py` в `.env` и подменить URL.
- [ ] Переключатель источника (фид / API) env-переменной, чтобы переход
      был обратимым; удалить фид-код после стабилизации.
- [ ] Синк должен не только добавлять, но и чистить: при успешном ответе
      API удалять будущие события, которых в ответе нет (перегенерация
      кеша сервиса — штатное дело, иначе фантомные строки останутся
      в БД навсегда). Прошедшие события не трогать.
- [ ] Миграция: при первом синке из API удалить строки со старыми
      uid in-the-sky (по формату uid, по аналогии с миграцией
      в `init_storage()`).
- [ ] Тексты: переключиться на `lang=ru`, обновить README.
      Атрибуцию In-The-Sky.org в приветствии и подвале дайджеста
      убирать только после полного отказа от фида; пока источники
      смешаны — она обязана оставаться.
- [ ] Недоступность API переживается как сейчас недоступность фида:
      бот отвечает из своей таблицы events, синк логирует и ждёт
      следующего запуска.

## Этап 8. Прогноз погоды к событиям (облачность и видимость)

Внизу сообщения о событиях — прогноз в местоположении юзера на время
наблюдения: облачность и видимость, чтобы оценить шансы что-то увидеть.

Источник — Open-Meteo (https://open-meteo.com): бесплатно для
некоммерческого использования, без API-ключа, почасовые `cloud_cover` и
`visibility`, прогноз до 16 дней, принимает таймзону параметром.
Лицензия CC-BY — атрибуция «Weather data by Open-Meteo.com» обязательна
(добавить рядом с атрибуцией событий).

- [x] Хранить координаты: `ALTER TABLE users ADD COLUMN lat/lon`
      (nullable, по образцу миграции в `init_storage()`); в `start.py`
      сейчас координаты выбрасываются после определения таймзоны —
      сохранять их в `add_user()`. Обновление — повторным «Share
      location»; выбравшим «Default time» и старым юзерам без координат
      погода просто не показывается.
- [x] `services/weather.py`: запрос почасового прогноза по (lat, lon,
      день, таймзона юзера), выборка `cloud_cover` и `visibility` на
      часы событий. Ошибки сети/API логируются, сообщение уходит без
      погодного футера (погода — украшение, не должна ломать ответ).
- [x] Кеш прогноза в памяти на ~1 час по ключу (округлённые координаты,
      день), чтобы листание недели не дёргало API на каждый клик.
- [x] Футер в `templates.py`: на каждое событие сообщения — строка
      «HH:MM: облачность N%, видимость M км» (формулировки — по месту);
      показывать только для будущих моментов в пределах горизонта
      прогноза (≤ 7 дней — дальше ненадёжно, 8–16 дней Open-Meteo
      отдаёт, но с оговорками; за пределами — футер опускается).
- [x] Подключить футер в хендлеры дневных сообщений: today / tomorrow /
      листалка недели / specific_date. Субботний дайджест — без погоды
      (это сводка на неделю вперёд, прогноз там бессмыслен).
- [x] Тесты офлайн: фикстура JSON-ответа Open-Meteo; выбор часов,

Нюанс на потом: время события — момент пика, а наблюдают обычно ночью;
v1 честно показывает прогноз на час события, ночное окно наблюдения —
возможное развитие (согласуется с фильтром видимости из этапа 6/7).

## Этап 9. Дайджест в таймзоне пользователя (на подумать)

Дневные хендлеры от серверной даты ушли: `get_day_message()` берёт
«сегодня» из профиля юзера (`timezones.today_in`). Субботняя рассылка —
нет: `send_weekly_digest()` считает окно от `date.today()`, и планировщик
там же проверяет `date.today().weekday() == SATURDAY`. У юзера в UTC+14
суббота наступает раньше серверной, поэтому и окно дайджеста, и
клавиатура под ним сдвинуты на день.

Просто подставить `today_in()` не выйдет: текст дайджеста строится один
раз на всех и рассылается циклом, а зависит он именно от даты. Варианты:

- [ ] Персональный дайджест: окно и текст на каждого юзера по его зоне.
      Точно, но N выборок и N рендеров на рассылку вместо одного, и
      будить планировщик придётся чаще раза в сутки — иначе «суббота»
      всё равно определяется по серверу.
- [ ] Группировка по зонам: одно сообщение на уникальную зону (их
      десятки, не тысячи) — компромисс между точностью и стоимостью.
- [ ] Оставить одно общее окно, но убрать зависимость от локали сервера:
      считать явно от UTC и сказать в тексте, что неделя — с субботы UTC.
      Дешевле всего; расхождение на день остаётся, но становится
      описанным поведением, а не случайным.

Сдвиг на день в сводке «на неделю вперёд» — не то же самое, что
неправильный «сегодня» в ответе на кнопку: цена ошибки низкая, так что
третий вариант может оказаться достаточным.

## Риски

- Фид неофициальный, без SLA: при недоступности — оставлять старые данные
  в БД и логировать, бот продолжает отвечать из кеша.
- Описания на английском: типовые события (фазы Луны и т.п.) позже можно
  локализовать шаблонами по ключевым словам summary.
